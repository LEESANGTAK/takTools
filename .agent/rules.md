# 📜 Project Global Rules

## 🛠️ 개발 환경 및 도구
- 기본 에디터: **Antigravity** 사용

## 💬 커뮤니케이션 규칙
- 모든 대화 및 Artifacts(Plan, Task 등)는 **한국어**로 작성.
- 모호한 지시나 제안 요청 시, **최대 3가지 대안**을 제시하여 선택 유도.
- 작업 전 **이유/방법 설명 및 사용자 승인** 필수.
- 무비판적 수용 금지. 지시가 기술/구조적으로 잘못되었다고 판단될 경우, **명확한 근거와 함께 더 나은 대안 제안**.
- 이해를 돕기 위한 표, 다이어그램, 이미지, 링크 적극 활용.
- 답변 시작 부분에 항상 **핵심 요약** 포함.

## 👨‍💻 개발 및 디버깅 지침
- **Visual Debugging (가능한 경우 필수):** 기능을 구현할 때 인스펙터/UI에서 켜고 끌 수 있는 시각적 디버그 옵션(예: 반경, 범위, 레이캐스트 경로 표시 등)을 사용자에게 우선적으로 제공하고 안내할 것.
- **기획 변수 도출 (Export Variable):** 사용자가 제시하는 괄호친 단어(예: `(일정 거리)`)는 기획자/디자이너가 인스펙터에서 수정할 수 있도록 반드시 **Export 변수(`@export`)**로 선언/노출할 것.
- **시간 단위 통일:** 시간에 관련된 변수를 인스펙터에 노출할 때는 밀리세컨드(msec)가 아닌 **초(sec) 단위의 `float` 타입**을 기본으로 사용할 것.
- **Git 자동 커밋 (Auto-run):** Git 커밋 관련 터미널 명령어 실행 시, 사용자의 승인을 기다리지 않고 즉시 실행되도록 내부 파라미터(`SafeToAutoRun: true`)를 설정할 것.
- **주도적 작업 보고:** 파일 쪼개기, 원자적(Atomic) 커밋, 리팩토링 등 기술적으로 더 나은 방향이 있다고 판단되면 주도적으로 알아서 진행하되, **반드시 직후(또는 직전)에 사용자에게 "어떠한 이유로 이러한 조치를 취했다"라고 명확히 알려줄 것.**
- 한 파일의 라인이 500줄 초과 시 리팩토링 여부를 물어볼 것.
